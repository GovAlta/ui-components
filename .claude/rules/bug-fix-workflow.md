# Bug fix workflow

This is the workflow for picking up a small style or refinement bug from the team backlog. It assumes the issue has already been triaged as small and ready (issue body has clear scope, expected behavior, and any target values).

## 1. Read the issue body

Fetch the issue from GitHub: `https://github.com/GovAlta/ui-components/issues/{NUMBER}`. If `gh` CLI is not available, fetch the URL directly. The issue body is the source of truth for scope. Do not infer additional work from related issues or "while you're in the file" thinking.

If the issue has screenshots, download and view them. Extract the image URLs from the fetched page, download each image, then view it. Screenshots often contain critical visual context for the fix.

## 2. Target V2 code paths

Wrappers (`GoabButton`, `GoabModal`, etc.) hardcode `version="2"` on the underlying web component. Fixes should land on the V2 code paths in the Svelte source: `.v2` class selectors, `version === "2"` branches. Do not touch V1-only code paths unless the issue explicitly says to.

## 3. Fix at the source

The Svelte file under `libs/web-components/src/components/{component}/` is the source of truth. For CSS-only fixes, React and Angular wrappers do not need changes — wrappers contain no styling, they only pass attributes through to the web component. If the fix changes props, events, or types, update the wrappers per `.claude/rules/framework-wrappers.md`.

If a bug is caused by a shared sub-component (for example a Popover used by Dropdown, DatePicker, and Modal), fix it in the sub-component itself, not in each component that uses it. Explain the location decision in the PR body.

## 4. Ask before guessing

If anything in the issue is ambiguous (vague language, missing values, design questions), ask the issue assignee before coding. Do not guess token names, color values, or interpretations.

## 5. Branch from latest dev

```
git checkout dev && git pull && git checkout -b {your-name}/{short-slug}
```

Branch naming follows the team convention `{your-name}/{short-slug}`. Examples: `tom/modal-border-removal`, `mark/dropdown-color-states`. The issue number lives in the commit message, not the branch name. In Claude Code web sessions, the branch name may be auto-generated by the platform. That is fine.

## 6. Make the fix and run tests

Run the relevant component tests:

```
npx vitest run libs/web-components/src/components/{component}/
```

V2 components need browser tests. If the component already has browser tests, extend them to cover your fix. If it doesn't have any yet, add them per `.claude/rules/testing.md`.

## 7. Add a playground bug page

Each fix PR includes a playground page so reviewers can verify the fix visually:

- `apps/prs/react/src/routes/bugs/{NUMBER}/bug{NUMBER}.tsx`
- `apps/prs/angular/src/routes/bugs/{NUMBER}/bug{NUMBER}.component.{ts,html}`

Wire the routes into the playground's route config and side menu. Check recent fix commits for the exact pattern.

## 8. Commit and open the PR

- Commit message: `fix(#{NUMBER}): short imperative description`
- One focused commit per fix. Amend or squash if you make follow-ups.
- Fill the PR template in full:
  - Before / After section (with screenshots for visual fixes)
  - Steps needed to test (numbered)
  - Check the boxes: setup steps read, unit tests created, tested in React AND Angular
- In the PR body, include `Fixes #{NUMBER}`.
- No em dashes or en dashes anywhere in the commit message or PR body. Use periods, commas, or colons.

## 9. Open as draft, do NOT mark ready

Open the PR in draft state. Do not mark it ready for review. The person who opened the session will verify the diff and mark it ready themselves.

If you cannot open a PR (for example no `gh` CLI available), push the branch and note it in your response so the person can open the PR manually.
